# 数据库迁移指南

本文档说明如何在已部署的项目中安全地升级数据库结构，而不影响现有数据。

## 🚀 快速开始（推荐方式）

### 方式一：命令行自动同步 ⭐

**最简单、最智能的方式！** 自动对比 `models.py` 和数据库结构，无需手动编写迁移代码。

```bash
# 1. 备份数据库（重要！）
cp data/app.db data/app.db.backup

# 2. 检查差异（不执行任何操作）
python app.py --update-db

# 3. 自动同步（会提示确认）
python app.py --update-db

# 4. 自动同步（跳过确认）
python app.py --update-db --yes
```

**输出示例：**

### 方式二：独立同步工具

如果不想启动应用，可以使用独立的同步工具：

```bash
# 检查差异
python db_sync.py

# 预览 SQL（不执行）
python db_sync.py --dry-run

# 自动同步
python db_sync.py --sync
```

## 工作原理

### 智能对比机制

系统会自动：

1. **读取 models.py**：解析所有 SQLAlchemy 模型定义
2. **检查数据库**：读取现有数据库的表结构
3. **智能对比**：找出差异（缺失的表、缺失的列、类型不匹配等）
4. **生成 SQL**：自动生成同步所需的 SQL 语句
5. **安全执行**：在事务中执行，失败自动回滚

### 检测的差异类型

- ✅ 缺失的表（models.py 有，数据库没有）
- ✅ 缺失的列（models.py 有，数据库没有）
- ⚠️ 多余的表（数据库有，models.py 没有）- 仅提示，不删除
- ⚠️ 多余的列（数据库有，models.py 没有）- 仅提示，不删除
- ✅ 列类型不匹配 - 提示但不自动修改

## 其他迁移方式

### 方式三：应用启动时自动创建（已内置）

应用启动时会自动调用 `db.create_all()`，创建缺失的表：

```python
# models.py 中的 init_db() 函数
def init_db(app):
    db.init_app(app)
    with app.app_context():
        db.create_all()  # 自动创建缺失的表
        _ensure_column(...)  # 添加缺失的列
        _ensure_backup_config()  # 初始化默认配置
```

**优点**：零配置，自动运行  
**缺点**：只能创建表，不能修改现有列

### 方式四：Flask-Migrate（专业方案）

适合大型项目，需要版本控制和回滚能力。

```bash
# 安装
pip install Flask-Migrate

# 初始化（仅首次）
flask db init

# 生成迁移脚本
flask db migrate -m "Add backup tables"

# 应用迁移
flask db upgrade

# 回滚
flask db downgrade
```

## 安全保障

### 数据安全

1. **非破坏性操作**：
   - 只添加缺失的表和列
   - 不会删除任何数据
   - 不会自动删除多余的表或列

2. **事务保护**：
   - 所有操作在事务中执行
   - 失败时自动回滚
   - 保证数据一致性

3. **确认机制**：
   - 默认需要手动确认
   - 使用 `--yes` 跳过确认（适合自动化部署）

### 最佳实践

1. **升级前必做**：
   ```bash
   # 备份数据库
   cp data/app.db data/app.db.backup_$(date +%Y%m%d_%H%M%S)
   
   # 备份整个 data 目录
   tar -czf data_backup_$(date +%Y%m%d_%H%M%S).tar.gz data/
   ```

2. **先检查后执行**：
   ```bash
   # 1. 先检查差异
   python app.py --update-db
   
   # 2. 确认无误后再执行
   # （第一次会预览 SQL，第二次输入 yes 执行）
   ```

3. **测试环境验证**：
   - 先在测试环境运行
   - 确认无误后再在生产环境执行

## 常见问题

### Q: 如何确认同步成功？

**A**: 查看输出信息：
```
✅ 数据库同步完成！
```

或者再次运行检查：
```bash
python app.py --update-db
# 应该显示：✅ 数据库结构与 models.py 完全一致，无需同步。
```

### Q: 同步失败怎么办？

**A**: 
1. 查看错误信息
2. 恢复备份：`cp data/app.db.backup data/app.db`
3. 检查数据库文件权限
4. 确保磁盘空间充足

### Q: 会删除多余的表或列吗？

**A**: 不会！系统只会：
- ✅ 添加缺失的表和列
- ⚠️ 提示多余的表和列（但不删除）

这样可以避免误删重要数据。

### Q: 可以修改现有列的类型吗？

**A**: 当前版本不支持自动修改列类型，因为这可能导致数据丢失。如需修改，请：
1. 手动编写 SQL
2. 或使用 Flask-Migrate

### Q: 如何在生产环境自动化部署？

**A**: 在部署脚本中添加：
```bash
#!/bin/bash
# 备份
cp data/app.db data/app.db.backup

# 更新代码
git pull

# 自动同步数据库（跳过确认）
python app.py --update-db --yes

# 重启服务
systemctl restart your-app
```

### Q: 与 Flask-Migrate 有什么区别？

**A**: 

| 特性 | 本工具 | Flask-Migrate |
|------|--------|---------------|
| 安装 | 无需额外依赖 | 需要安装 |
| 使用 | 一条命令 | 多个步骤 |
| 版本控制 | 无 | 有 |
| 回滚 | 不支持 | 支持 |
| 学习成本 | 低 | 中等 |
| 适用场景 | 小型项目 | 大型项目 |
